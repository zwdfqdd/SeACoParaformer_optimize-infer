# SeACo-Paraformer 推理服务性能压测报告

本报告系统记录推理服务在 **NVIDIA A10（24GB）+ 256 核 + 512GB** 环境下的性能压测。
结论前置（见「核心结论速览」），后续章节按物理顺序为：
二 线程池基准 → 三 字级时间戳开销 → 四 WORKERS 显存换吞吐 → 五 Faiss 与模块组合 →
六 音频时长 → 七 各精度/后端横向对比 → 八 系统性结论。
其中**三、五同属「功能模块开销」测试**（见下方说明）。

> 说明：字级时间戳（三）与 Faiss/模块组合（五）同属「功能模块开销」测试。第三章为
> WORKERS=10 的时间戳开/关同口径对照；第五章为 WORKERS=11 的全关基线 + 各模块（纯时间戳/
> Faiss）单项开销对照。两章口径各自自洽，避免跨 WORKERS 混比。

---

## 核心结论速览（TL;DR）

> 测试设备：A10 24GB + 256 核 + 512GB 内存，`trt_fp16`，30s 音频，充分并发压测。

### 涉及参数说明

| 参数 | 含义 |
|------|------|
| `WORKERS` | uvicorn worker 进程数；多进程隔离 CPU 竞争，受 GPU 显存约束 |
| `CPU_THREAD_POOL_SIZE`（表中简作 CPU_POOL） | CPU 流水线线程池大小（Stage1 VAD + Stage2 特征提取），per-worker |
| `VAD_SESSION_POOL_SIZE`（表中简作 VAD_POOL） | Silero VAD ORT session 池大小（round-robin 并行），per-worker |
| `BATCH` | GPU 动态合批的最大 batch size |
| `BATCH_TIMEOUT` | 动态合批的最大排队等待（ms），工业标准 max_queue_delay |
| `GPU_STREAM_POOL_SIZE` | TRT engine 多 stream 多 execution_context 池（榨干 GPU sm 利用率）|
| `MODEL_PRECISION` | 推理精度，本报告固定 `trt_fp16`（TensorRT fp16）|
| `ENABLE_WORD_TIMESTAMP`（表中简作「时间戳」） | 字级时间戳开关；开启加载第 5 段 timestamp engine（含 BLSTM）|
| `ENABLE_FAISS_CORRECTION`（表中简作「FaissB」） | 默认词表 Faiss 后处理纠错开关（路径 B）|
| `ENABLE_HOTWORD` | 客户端热词 SeACo 在线增强开关（路径 A）|
| `OMP/MKL/OPENBLAS_NUM_THREADS` | 底层数学库线程数；固定 1（防 libgomp 高并发崩溃）|
| 吞吐(audio_s/s) | 每秒可处理的音频秒数 = QPS × 单条音频时长，衡量总处理能力 |
| QPS | 每秒完成的请求数 |

### 最优参数矩阵（按业务场景选一行）

| 业务场景 | WORKERS | CPU_POOL | VAD_POOL | 时间戳 | FaissB | 峰值吞吐(audio_s/s) |
|---------|:-------:|:--------:|:--------:|:------:|:------:|:-------------------:|
| **纯转写 · 极限吞吐** | 11 | 32 | 4 | false | false | **~2933** |
| **通用识别 · 默认词表纠错**（★推荐默认） | 11 | 32 | 2 | false | true | ~2559 |
| **需要字幕 / 字级时间戳** | 10 | 16 | 1~2 | true | 按需 | ~1970 |

> 全程固定：`BATCH=12` / `BATCH_TIMEOUT=10` / `GPU_STREAM_POOL_SIZE=4` /
> `OMP_NUM_THREADS=MKL=OPENBLAS=1`。

### 五条关键发现

1. **`CPU_THREAD_POOL_SIZE` 是第一大杠杆**：1→16 吞吐提升 7.4x，是唯一的吞吐主导参数；
   拐点在 16（256 核 / WORKERS 的每 worker 不超订分界），16→32 仅 +3%。
2. **`VAD_SESSION_POOL_SIZE` 影响很小**：全程波动 <5%，2~4 最优，内存紧可降到 1~2。
3. **显存决定 WORKERS**：关闭时间戳省 ~1GB engine 显存，可开到 11（比 10 再 +5%）；
   开启时间戳需回落到 10 防 OOM。
4. **功能模块开销**（相对纯 ASR 基线 2878，同 WORKERS=11）：字级时间戳 **-32%**（硬瓶颈，
   开启后吞吐钉死在 ~1950），默认词表 Faiss 后处理 -11%；两者叠加不额外累加（时间戳已封顶）。
   热词路径 A（SeACo 在线）开销**未测**（压测均不带 hotwords，路径 A 未触发）。
5. **吞吐随音频时长升高**：长音频（≥30s）吞吐最佳（121s 达 3187），短音频（<10s）
   受请求级固定开销拖累降到 ~1800；压测/切段基准取 15~30s 最有代表性。

### 应用建议

线上默认用「通用识别」行（WORKERS=11 / CPU_POOL=32 / VAD_POOL=2，仅开 Faiss）；
需要字幕再单独开时间戳并把 WORKERS 降到 10；参数随硬件（核数 / 显存）变化需按
第一章脚本重跑网格重新定位拐点。完整决策流程见第八章；各精度/后端横向对比见第七章。

> 纯转写追求吞吐 → 关闭（默认，`words` 为空）。

### 3.4 瓶颈根因分析（为何时间戳是硬瓶颈）

阅读时间戳计算图（`seaco_paraformer/predictor.py` 的 `get_upsample_timestamp`）可定位根因。
核心三步：

```python
context = hidden.transpose(1, 2)              # (B, D, T),  D=512
output2 = self.upsample_cnn(context)          # ConvTranspose1d → (B, D, T×3)  序列 ×3
output2, _ = self.blstm(output2.transpose(1,2))  # 双向 LSTM 在 T×3 上跑 → (B, T×3, 1024)
alphas2 = sigmoid(self.cif_output2(output2))  # Linear 1024→1
us_cif_peak = _cif_wo_hidden_v1(us_alphas, ...)  # cumsum 前缀扫描
```

开销按影响排序：

1. **双向 BLSTM 是绝对主因（GPU 痛点）**：`nn.LSTM(512,512,bidirectional=True)` 跑在
   `T×upsample_times` 长度上。主力桶 67 帧 × 3 = **201 个时间步**，LSTM **严格串行**
   （每步依赖前一步隐状态），且双向要正反各跑一遍。GPU 靠大规模并行取胜，串行 RNN
   让上千 CUDA core 大部分时间空等；TensorRT 对 LSTM 的优化本就有限，fp16 加速打折。
   对比主链路：encoder 是 Transformer（self-attention 全并行）、decoder 是并行 matmul，
   **唯独 timestamp 引入串行 RNN**，直接拉低整段 GPU 吞吐。
2. **ConvTranspose1d 上采样 ×3**：序列从 T 扩到 T×3，后续 blstm/linear 全在 3 倍长度上
   算，计算量与激活显存均 ×3。
3. **`_cif_wo_hidden_v1` 的 cumsum**：前缀依赖（scan），GPU 并行度有限，但逐元素轻量，
   相比 BLSTM 可忽略。

**为何不能靠"合并进 CIF"优化**：实测「时间戳+Faiss」(1970) 与「纯时间戳」(1948) 几乎相等、
且都钉在 ~1950，证明瓶颈是 timestamp 内部的**计算量（BLSTM）**，而非"多一段 engine"的
串联/调度开销。合并进 CIF 无法减少 BLSTM 那 201 步串行计算（一分不少），反而会牺牲
**「不启用时零成本」**这一分离架构的最大价值——`ENABLE_WORD_TIMESTAMP=false` 时完全不跑
BLSTM，吞吐回到 2878；若并入 CIF 则每次推理都背着 -32% 开销、无法关闭。**故维持独立
第 5 段 engine 的设计。**

**真正的优化方向**（如需降低这 32%，须从算子本身入手）：
- 降 `upsample_times`（3→2 或 1）：序列变短，BLSTM 步数等比下降；代价是时间戳精度变粗
  （~20ms → ~30ms）。
- BLSTM 换全并行结构（TCN / 轻量 self-attention）：需改模型 + 重训/蒸馏，工程量大。
- 接受现状：字幕场景通常不追求极限吞吐，-32% 可接受；纯转写关闭即可（当前默认）。

   WORKERS 需回落到 10，否则有 OOM 风险。

### 5.4 峰值吞吐与压测并发的关系

> 峰值吞吐需更高并发才能榨出，120 并发对拐点档偏保守。

## 七、各精度 / 后端横向对比（各自最大 WORKERS）

比较各推理精度/后端在"该精度显存能容纳的最大 WORKERS"下的吞吐上限。

### 7.1 复现环境与固定参数

| 项 | 值 |
|----|----|
| 硬件 | A10 24GB + 256 核 + 512GB |
| 压测脚本 | `tests/test_service.py` |
| 并发 × 请求数 | 120 × 3000 |
| 测试音频 | `test_data/audio_16000_30s.wav`（16kHz 单声道，30s） |
| BATCH / BATCH_TIMEOUT | 12 / 10ms |
| CPU_THREAD_POOL_SIZE | 0（自动全核=256） |
| VAD_SESSION_POOL_SIZE | 4 |
| GPU_STREAM_POOL_SIZE | 4（仅 TRT 后端） |
| OMP/MKL/OPENBLAS | 1/1/1 |
| ENABLE_WORD_TIMESTAMP | false（关闭字级时间戳） |
| ENABLE_HOTWORD / ENABLE_FAISS_CORRECTION | 均 true（默认）；★压测请求**不带 hotwords**，故路径 A 未触发，实际仅默认词表 Faiss（路径 B）生效 |

> 注：本章各组用各精度**自身最大 WORKERS**（受显存约束），CPU_POOL/VAD_POOL 未按第二章
> 拐点调优（用默认 0/4），故 trt 数值低于第五章调优后的峰值，仅作**跨精度横向对比**，
> 不代表各精度的调优上限。

### 7.2 启动命令（逐组，容器内 /app 下）

```bash
# trt_fp32（最大 7 worker，fp32 分段权重最大）
MODEL_PRECISION=trt_fp32 WORKERS=7  bash run.sh
# onnx_fp32（ORT/CUDA，最大 12 worker）
MODEL_PRECISION=onnx_fp32 WORKERS=12 bash run.sh
# pt（PyTorch 原生 GPU，最大 10 worker）
MODEL_PRECISION=pt WORKERS=10 bash run.sh
# onnx_int8（ORT/CPU，10 worker）
MODEL_PRECISION=onnx_int8 WORKERS=10 bash run.sh
# 压测（另一终端）
python tests/test_service.py --audio test_data/audio_16000_30s.wav --concurrency 120 --total 3000
```

### 7.3 结果

| 精度 | 后端/设备 | 最大 WORKERS | 总耗时(s) | 平均延迟(ms) | P99(ms) | QPS | 吞吐(audio_s/s) | 成功率 |
|------|-----------|:-----------:|----------:|-------------:|--------:|------:|----------------:|:------:|
| **trt_fp16**※ | trt/cuda | 11 | ~31 | ~1483 | ~2973 | ~94 | **~2933** | 100% |
| trt_fp32 | trt/cuda | 7 | 64.10 | 2498 | 5214 | 46.80 | 1407 | 100% |
| onnx_fp32 | ort/cuda | 12 | 86.84 | 3394 | 6718 | 34.55 | 1039 | 100% |
| pt | pt/cuda | 10 | 184.82 | 7143 | 22459 | 16.23 | 488 | 100% |
| onnx_int8 | ort/cpu | 10 | 2483 | 51943 | 296362 | 0.99 | 30 | **81.5%（554 失败）** |

※ trt_fp16 非本批次数据，引用第五章 WORKERS=11 调优峰值（150 并发口径），仅作对照标杆；
其余四组为本次同口径（120 并发 × 3000，默认 CPU_POOL/VAD_POOL）实测。

> 最大 WORKERS 受显存约束（单份权重体积 + arena/allocator 策略）：
> onnx_fp32(12) > trt_fp16(11) > pt(10) > trt_fp32(7)。
> onnx_fp32 能开 12 因 ORT arena 改 kSameAsRequested 后按需分配（见代码优化记录）。

### 7.4 结论

1. **trt_fp16 是绝对最优**：~2933 audio_s/s，是 trt_fp32 的 2x、onnx_fp32 的 2.8x、
   pt 的 6x。fp16 权重小 → 可开 11 worker + engine 高度优化。
2. **trt_fp32 吞吐 1407**：虽单卡精度无损，但 fp32 权重大只能开 7 worker，且无 fp16 加速，
   吞吐仅 trt_fp16 的一半。仅无损基线用途。
3. **onnx_fp32(GPU) 1039**：ORT 整体模型 + CUDA，能开 12 worker（arena 按需分配后显存友好），
   但 ORT 图优化/调度不及 TRT，吞吐低于 trt_fp32。CPU 兜底/无 TRT 环境可用。
4. **pt(GPU) 488**：PyTorch 原生无 dynamic batching 的 GPU 池化，单模型串行，
   吞吐最低。定位验证/无 TRT 环境，不适合高并发生产。
5. **onnx_int8(CPU) 崩溃级**：QPS 0.99、554 失败、P99 达 296s——CPU int8 在 120 并发下
   完全过载。int8 动态量化仅适合**低并发 CPU 部署**，切勿高并发压测。

### 7.5 选型建议

| 场景 | 推荐 | 理由 |
|------|------|------|
| GPU 高并发生产 | **trt_fp16 / trt_int8_enc** | 吞吐最高，显存可控 |
| GPU 无损基线 | trt_fp32 | 精度无损，吞吐减半 |
| 无 TRT 的 GPU 环境 | onnx_fp32 | 免转换，吞吐中等 |
| 精度验证 / 调试 | pt | 免转换，直接加载权重 |
| 低并发 CPU 部署 | onnx_int8 | 体积-75%；★严禁高并发 |

## 八、系统性结论与应用建议

测试设备：A10 24GB + 256 核 + 512GB 内存，`trt_fp16`。

### 8.1 参数影响力排序（从大到小）

| 排名 | 参数 | 影响幅度 | 拐点/最优 | 说明 |
|:----:|------|---------|-----------|------|
| 1 | `CPU_THREAD_POOL_SIZE` | 7.4x（1→16） | **16**（16→32 仅 +3%）| 唯一吞吐主导；Stage1/2 CPU 处理能力 |
| 2 | 功能模块（时间戳/Faiss） | 时间戳 -32%（硬瓶颈），Faiss -11%（热词 A 未测）| 按需开 | 时间戳开销最大 |
| 3 | `WORKERS`（受显存约束） | +5%（10→11）| 关时间戳 11 / 开 10 | 多进程隔离 CPU 竞争 |
| 4 | 输入音频时长 | 短音频吞吐降 ~40% | 15~30s 最佳 | 请求级固定开销摊薄 |
| 5 | `VAD_SESSION_POOL_SIZE` | <5%（噪声级）| 2~4 | 影响最小，内存紧可降 |

### 8.2 最优配置矩阵（实测峰值，30s 音频，充分并发）

| 场景 | 时间戳 | FaissB | WORKERS | CPU_POOL | VAD_POOL | 峰值吞吐(audio_s/s) |
|------|:------:|:------:|:-------:|:--------:|:--------:|-------------------:|
| **纯转写 · 极限吞吐** | false | false | 11 | 32 | 4 | **~2933** |
| **通用识别**（★线上默认） | false | true | 11 | 32 | 2 | ~2559 |
| **需要字级时间戳** | true | 按需 | 10 | 16 | 1~2 | ~1970 |

> 热词路径 A（SeACo 在线，`ENABLE_HOTWORD=true` 且请求带 hotwords 时触发）的吞吐
> **未测**——本报告所有压测均不带 hotwords。开启会增加 bias 编码开销，需另行实测。

### 8.3 选型决策流程

1. **是否需要字幕/对齐/精确定位？**（时间戳开销最大，约 -32%，务必确认必要性）
   - 是 → 开 `ENABLE_WORD_TIMESTAMP=true`，`WORKERS=10`（防显存 OOM），`CPU_POOL=16`
     （开启后吞吐钉在 ~1950，再大线程无用）；
   - 否 → 关时间戳，`WORKERS=11`，`CPU_POOL=32`（榨干峰值）或 16（省一半线程仅损 3%）。
2. **热词需求？**
   - 客户端会传热词 → `ENABLE_HOTWORD=true`（路径 A 吞吐开销未测，需带 hotwords 实测）；
   - 依赖服务端默认词表纠错 → `ENABLE_FAISS_CORRECTION=true`（约 -11% 吞吐）；
   - 纯通用转写、追求极限 → 两者全关。
3. **`VAD_POOL` 取 2**（全场景稳妥）；`BATCH_TIMEOUT/BATCH/GPU_STREAM_POOL_SIZE` 保持默认。

### 8.4 迁移到其他硬件的注意事项

- 参数为 **per-worker**，实际总线程 ≈ `WORKERS × CPU_POOL`；不超订分界 ≈ `核数 / WORKERS`。
  本环境 256 核 / 11 worker ≈ 23/worker，故 CPU_POOL 到 32 已接近上界。
  **换硬件（核数/显存不同）务必按第一章 `tests/benchmark_grid.sh` 重跑网格重新定位拐点**。
- 显存更大（A100 40/80GB）可继续上调 WORKERS；显存更小（2080Ti 11GB）建议 WORKERS=1
  单进程模式（见 `docs/DEPLOY.md` 模式 A）。
- **稳定性红线**：`OMP_NUM_THREADS=MKL_NUM_THREADS=OPENBLAS_NUM_THREADS=1` 必须固定，
  否则 libgomp 在高并发下崩溃（详见 `docs/DEPLOY.md`）。

### 8.5 稳定性

- 全部压测组（时间戳开/关网格 + 纯时间戳网格各 24 组 + WORKERS=11 组 + 时长扫描 9 组）
  **0 失败**，服务在 150 并发下稳定。

## 九、句子级时间戳（CT-Transformer 标点分句）专项开销

**目的**：量化句子级时间戳（`ENABLE_SENTENCE_TIMESTAMP`，CT-Transformer 标点模型）相对
字级时间戳的额外开销，并暴露一个关键的线程配置陷阱。

**配置**：A10 24GB，`trt_fp16`，30s 音频，120 并发 × 3000 请求，`CPU_POOL=32`/`VAD_POOL=2`
（run.sh 默认）。三组递进对照。

### 9.1 三组对照结果

三组均 120 并发 × 3000 请求；★三组的 `ENABLE_FAISS_CORRECTION` 均为**默认 true**
（请求不带 hotwords，故仅默认词表 Faiss 路径 B 生效，路径 A 未触发）。递进只加时间戳开关：

| 组 | 配置（均含默认 Faiss） | WORKERS | QPS | 吞吐(audio_s/s) | 平均延迟 | P99 | GPU 利用率 | CPU 利用率 |
|----|------|:-------:|:---:|:---------------:|:--------:|:---:|:----------:|:----------:|
| 1 默认（无时间戳） | Faiss | 11 | 69.89 | 2101 | 1667ms | 4070ms | 79.1% | 16.2% |
| 2 +字级时间戳 | Faiss+word_ts | 10 | 62.03 | 1865 | 1876ms | 4251ms | 97.1% | 19.9% |
| 3 +字级+句子级 | Faiss+word_ts+sentence（intra_op=1） | 10 | 61.59 | 1852 | 1884ms | 4337ms | 97.3% | 20.5% |

> 参照（同环境、同 120 并发、**全关模块**纯转写，`ENABLE_HOTWORD=false`+`ENABLE_FAISS_CORRECTION=false`）：
> 吞吐 **2866** audio_s/s，GPU 96.7%，CPU 22.9%。可见组1（2101）低于全关基线，
> 差额即默认 Faiss 后处理开销（与第五章 5.3「仅 Faiss」-11% 一致）。
>
> 本章新增 GPU + CPU 利用率监测（`tests/test_service.py` 后台 nvidia-smi + /proc/stat 采样）。

### 9.2 结论

1. **句子级 CT 近乎零开销**（组3 vs 组2）：吞吐 1865→1852（**-0.7%**，测量噪声内），
   GPU/CPU/显存几乎不变。因 CT 标点推理在 CPU 单线程跑（几 ms/请求），而 GPU 已是瓶颈
   （97%），CPU 仅 20%，句子级被流水线完全吸收。

2. **字级时间戳开销 -11%**（组2 vs 组1，均含默认 Faiss、120 并发口径）：GPU 79%→97% 打满，
   叠加 WORKERS 11→10（显存约束）。与第三/五章 -32% 的差异源于口径不同（本章两组均含
   Faiss 且 120 并发；第五章为扣除 Faiss 的纯时间戳单项、150 并发全关基线对照）。

3. **CT 标点 CPU 占用极低**：句子级开启后 CPU 仅从 19.9%→20.5%，印证 CT 单线程 + 
   结果合并走 CPU 线程池不阻塞主链路。

### 9.3 ★关键教训：CPU 侧 ORT session 必须限线程（否则高并发反噬）

初版 CT session **未设 `intra_op_num_threads`**，ORT CPU EP 默认取物理核数（256）。
多 worker（10）× 高并发（120）下，每个分句推理都想开满核 → **线程超额订阅** →
CPU 打满、GPU 被饿死。实测对比（组3 修复前后）：

| 组3 | intra_op | QPS | 吞吐 | 平均延迟 | GPU | CPU |
|-----|:--------:|:---:|:----:|:--------:|:---:|:---:|
| 修复前 | 默认(=核数) | 27.26 | 820 | 4315ms | 52.7% | **90.3%** |
| 修复后 | **1** | 61.59 | 1852 | 1884ms | **97.3%** | 20.5% |

修复（`PUNC_INTRA_OP_THREADS=1`，与 VAD 同策略）后吞吐 **+126%**，CPU 回落、GPU 重新喂饱。

**通用原则**：CPU 侧推理 session（VAD / CT 标点 / ORT CPU 主模型）在多 worker × 高并发
部署下**必须显式限制 intra_op 线程数**，并发靠 worker 进程 + 请求并发提供，不靠 session
内多线程——否则线程超额订阅会严重反噬吞吐。

### 9.4 应用建议

- 句子级时间戳（需按句/子句展示的场景，如字幕、纪要）可**放心开启**——相对字级时间戳
  几乎不增加开销（前提：字级时间戳已开，句子级强依赖它）。
- 保持 `PUNC_INTRA_OP_THREADS=1`（默认）；勿调大。
