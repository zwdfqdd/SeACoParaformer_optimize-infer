# CPU 线程池参数网格压测报告

本报告记录 `CPU_THREAD_POOL_SIZE` × `VAD_SESSION_POOL_SIZE` 的网格压测结果，
用于确定 GPU/TRT 部署下 CPU 侧线程池参数的最优取值。

---

## 一、测试目的

在 TRT（GPU）后端下，只有两个 CPU 侧参数会影响吞吐：

| 参数 | 作用 | 说明 |
|------|------|------|
| `CPU_THREAD_POOL_SIZE` | Stage1 VAD + Stage2 特征提取的线程池大小 | 对所有后端生效 |
| `VAD_SESSION_POOL_SIZE` | Silero VAD ORT session 池大小（round-robin） | VAD 单 session 线程数硬编码为 1，靠多 session 并行 |

> 注：`ORT_INTRA_OP_THREADS / ORT_INTER_OP_THREADS` 仅作用于「主 ASR 走 CPU 后端
> （onnx_fp32/onnx_int8）」，TRT/GPU 部署下完全不生效，故不纳入本网格。详见
> `docs/DEPLOY.md` CPU 侧线程参数说明。

本次目标：在两参数的笛卡尔积上定位吞吐拐点，给出生产推荐配置。

```bash
# 容器内 /app 下，一键跑完 24 组（每组自动启停服务，避免显存累积爆炸）
bash tests/benchmark_grid.sh

# 结果输出：
#   - 终端：汇总表
#   - 文件：benchmark_grid_result.csv（本报告原始数据来源）
```

网格脚本执行逻辑（每组）：
启动 `run.sh`（注入变量）→ 轮询 `/health` 就绪 → `test_service.py` 压测 →
解析结果 JSON → 杀服务 → 冷却 20s → 下一组。全部跑完输出汇总表 + CSV。

单组配置扩展：编辑 `tests/benchmark_grid.sh` 的 `CONFIGS` 列表（每行 `"CPU_POOL VAD_POOL"`）。

## 五、分析

### 5.1 各 CPU_POOL 档位最优吞吐

| CPU_POOL | 最优 VAD_POOL | 吞吐(audio_s/s) | QPS | 相对 CPU_POOL=1 |
|---------:|--------------:|----------------:|------:|----------------:|
| 1  | 1 | 320  | 10.65 | 基准 |
| 2  | 2 | 594  | 19.75 | 1.9x |
| 4  | 4 | 1032 | 34.34 | 3.2x |
| 8  | 8 | 1253 | 41.69 | 3.9x |
| 16 | 8 | 2356 | 78.37 | 7.4x |
| 32 | 1 | **2429** | **80.80** | **7.6x** |

### 5.2 关键结论

1. **`CPU_THREAD_POOL_SIZE` 是绝对主导因素**：从 1 提到 16，吞吐提升 7.4x，是唯一有效大杠杆。
   说明 Stage1/2（VAD + fbank）的 CPU 处理能力是默认小线程池下的瓶颈。

2. **拐点在 16，16→32 收益急剧衰减**：吞吐 2356 → 2429，仅 +3%。
   此时 GPU（Stage3）已成为新瓶颈，CPU 侧继续加线程无意义。

3. **`VAD_SESSION_POOL_SIZE` 影响很小**：CPU_POOL ≥ 8 后，VAD_POOL 取 1/2/4/8 的吞吐
   波动都在 ±5% 内（噪声级），无单调规律。低 CPU_POOL 区个别点（如 1,2 的 214）为压测抖动，
   不代表趋势。

4. **全部 24 组 0 失败**，服务在 120 并发 × 2500 请求下稳定。

### 5.3 生产推荐配置

| 目标 | CPU_THREAD_POOL_SIZE | VAD_SESSION_POOL_SIZE | 说明 |
|------|:--------------------:|:---------------------:|------|
| **性价比最优（推荐）** | 16 | 4 | 拐点位置，比 32 省一半线程仅损 3% 吞吐 |
| 极限吞吐 | 32 | 1 | 榨干 CPU，实测峰值 2429 audio_s/s |
| 显存/内存受限 | 16 | 2 | VAD_POOL 降到 2 省内存，吞吐几乎不变 |

> 注：以上基于 256 核 + `WORKERS=10`。参数为 per-worker，实际总线程 ≈ WORKERS × 值。
> 不超订分界 ≈ 核数 / WORKERS ≈ 25/worker，故 CPU_POOL 到 32 已接近上界，再大将超订降速。
> 不同核数 / WORKERS 组合请重跑本网格重新定位拐点。
